# パイプライン説明サマリー（面接用）

## 1. プロジェクト全体像

**目的**: 動画に映る人間（例: 野球バッターのスイング）からモーションデータを抽出し、BVH/FBX/JSON形式の3Dデータとして出力するPythonツール。

**本番環境**: AWS EC2（GPUインスタンス）

## 2. 4ステージパイプライン

```
動画(MP4等) → ①フレーム抽出 → ②2Dポーズ推定 → ③データ加工 → ④3D変換・出力
                 CPU/OpenCV      GPU/MMPose       CPU/SciPy      GPU/VideoPose3D
```

| ステージ | 処理内容 | キーポイント |
|---|---|---|
| **VideoExtractor** | 動画→フレーム画像列（指定FPSで抽出） | OpenCV、フレーム数保存の保証 |
| **PoseEstimator** | フレーム→2D関節座標（COCO 17点） | MMPose HRNet、バッチ処理、GPU OOM時の自動リトライ（バッチ半減） |
| **DataProcessor** | 欠損補完・スムージング・関節削除・角速度算出 | スプライン補間、信頼度閾値(0.3)ベース、元データ不変の保証 |
| **Converter3D** | 2D→3D変換、座標系変換、フォーマット出力 | VideoPose3D（Temporal CNN）、COCO→H36M関節マッピング、クォータニオン正規化 |

## 3. 設計判断のポイント

### GPUメモリ管理
OOM発生時にバッチサイズを自動半減して再試行。最小でも失敗なら`GPUMemoryError`を送出。本番環境でのGPU安定運用を考慮。

### 関節マッピング（COCO→H36M）
2D推定のCOCO 17関節から、3Dモーション標準のHuman3.6M 17関節へ変換。直接対応するもの（肩・肘など）とミッドポイント合成（Hip=左右腰の中点、Spine=肩腰の中点）を組み合わせ。

### データ補完戦略
信頼度が閾値以下のフレームをCubicSplineで補間。補間データには`INTERPOLATED_CONFIDENCE=0.5`マーカーを付与し、元の有効データは一切変更しない（深いコピーで保護）。

### 出力フォーマット対応
- **BVH Position**: Blender互換のワールド座標出力（X軸反転でZ-up変換対応）
- **BVH Hierarchical**: H36Mスケルトン階層＋ZXYオイラー角
- **JSON**: 右手系Y-up、階層情報付き
- **FBX**: ASCII形式のスケルタルデータ

## 4. 5つの正当性プロパティ

1. **フレーム数保存** — 入力フレーム数 = 出力フレーム数
2. **関節データ完全性** — 全フレームで全関節のデータが存在
3. **時間的一貫性** — 角速度が全て有限値、出力長 = 入力-1
4. **信頼度の単調性** — 補完処理で元の有効データが変わらない
5. **座標系の一貫性** — クォータニオンが正規化済み（ノルム≈1）

pytest + hypothesisのプロパティベーステストで検証。

## 5. 技術スタック

| ライブラリ | 用途 |
|---|---|
| **MMPose** (HRNet) | 2Dポーズ推定 |
| **VideoPose3D** | 2D→3D変換（Temporal CNN、受容野243フレーム、残差ブロック8層） |
| **OpenCV** | 動画デコード・フレーム抽出 |
| **SciPy** | CubicSpline補間、ガウシアンフィルタ |
| **PyTorch + CUDA** | GPU推論基盤 |

## 6. 想定Q&A

| 想定質問 | 回答の方向性 |
|---|---|
| なぜMMPose？ | HRNet系の高精度モデルが豊富、APIが整備、商用利用可 |
| 2D→3D変換の原理は？ | VideoPose3Dは時系列の2D関節座標をTemporal CNNに通し、前後フレームの文脈から奥行きを推定 |
| 節の数が違うボーンへの変換は？ | COCO(17)→H36M(17)のマッピングテーブル＋ミッドポイント合成で実装 |
| GPU OOM対策は？ | バッチサイズ自動半減リトライ。最小バッチでも失敗なら明示的エラー |
| BVHフォーマットの知見は？ | 階層構造定義（HIERARCHY）＋フレームデータ（MOTION）。Position modeとRotation modeの2方式を実装 |
| テスト戦略は？ | GPU依存部はモック、5つの不変条件をhypothesisでプロパティテスト |
